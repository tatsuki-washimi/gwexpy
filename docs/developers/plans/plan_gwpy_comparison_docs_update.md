# GWpy比較ドキュメント更新計画

**作成日**: 2026-01-29
**ステータス**: 提案中 (Proposed)

## 1. 目的とゴール
ユーザーからの問い合わせにより発見された `gwexpy_for_gwpy_users_ja.md` (GWpyユーザー向けガイド) の内容をレビューし、最新の実装状態に合わせて更新する。
また、`detailed_gap_analysis_ja.md` (GUIギャップ分析) の知見を適切に反映またはリンクさせ、ユーザーが機能差異を正しく理解できるようにする。

## 2. フェーズ別詳細ロードマップ

### Phase 1: 現状分析と整合性チェック
*   **タスク**:
    *   `docs/web/ja/guide/gwexpy_for_gwpy_users_ja.md` を読み込む。
    *   記載されている機能（特に `FrequencySeriesList`, `SeriesMatrix`, `Fields` など）が最新のコードベースに存在し、APIが変更されていないか確認する。
    *   `devel/reports/detailed_gap_analysis_ja.md` の内容を要約する。

### Phase 2: ドキュメント更新
*   **タスク**:
    *   `gwexpy_for_gwpy_users_ja.md` に以下の更新を行う:
        *   最新の機能追加（もしあれば）を追記。
        *   GUI (`pyaggui`) に関する記述を修正し、現状の制限事項（ギャップ分析の結果）を注記として追加する。
        *   GWpyとの互換性に関する明確な記述（どのバージョンをベースにしているか等）を追加。
    *   必要であれば、英語版のドキュメント作成または更新も検討する（今回は日本語版優先）。

### Phase 3: 検証
*   **タスク**:
    *   ドキュメントのビルド (`build_docs` スキル) を行い、警告が出ないことを確認する。
    *   リンク切れがないか確認する。

## 3. テスト・検証計画
*   **ドキュメントビルド**: `sphinx-build` によるHTML生成確認。
*   **内容確認**: 生成されたHTMLを目視確認し、レイアウトと記述の整合性をチェック。

## 4. 推奨モデル・スキル・工数見積もり

### 推奨モデル
*   **Gemini 3 Flash**: ドキュメントのテキスト処理とコードの存在確認が主であるため、高速で低コストなモデルが最適。

### 推奨スキル
*   `analyze_code`: 機能の裏付け調査に使用。
*   `sync_docs`: ドキュメント更新と整合性維持に使用。
*   `build_docs`: 変更後の検証に使用。

### 工数見積もり
*   **予想所要時間**: 30分
*   **クオータ消費**: Low (テキスト編集主体)
