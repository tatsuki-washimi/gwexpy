# gwexpy 機能・ドキュメント・チュートリアル ギャップ分析レポート

生成日時: 2026-03-31
ステータス: 調査完了

## 1. 概要
本レポートは、`gwexpy` リポジトリのコードベース、テストスイート、および既存のチュートリアルを比較調査し、ユーザーが機能を最大限に活用するために不足しているリソース（ドキュメント、チュートリアル、未実装機能）を特定したものである。

## 2. 既存機能だがチュートリアル・ドキュメントが完全に欠落している項目 (P0-P1)
以下の機能は、`tests/` において網羅的なテストがパスしており、コードとしては成熟しているが、ユーザー向けのガイドが一切存在しない。

### 2.1 ヒストグラム解析フレームワーク (`gwexpy/histogram/`)
- **詳細**: `tests/histogram/` 内に `test_histogram_core.py` から `test_histogram_io.py` まで、シリアライゼーション（HDF5/JSON）、再ビン化、ROOT連携等のテストが完備されている。
- **欠落内容**: `intro_histogram.ipynb` 等の入門ガイド。多次元ヒストグラムの構築や、統計・可視化の基本的な使い方の説明。
- **優先度**: **高 (P0)**

### 2.2 空間場解析 (Vector / Tensor Field)
- **詳細**: `ScalarField` のチュートリアルはあるが、`VectorField`（磁力線等）や `TensorField`（歪等）の扱いが説明されていない。
- **欠落内容**: `VectorField` を用いたベクトル場の可視化、空間相関解析 (`coherence_map`, `time_delay_map`)。
- **優先度**: **中 (P1)**

### 2.3 コマンドライン・インターフェース (CLI)
- **詳細**: `gwexpy/cli/` にはデータ取得、変換、簡易プロット等のコマンドが実装されている。
- **欠落内容**: CLI のコマンド一覧、引数の説明、ターミナルからの利用例。
- **優先度**: **高 (P1)**

### 2.4 高度な時系列分解・前処理
- **詳細**: `TimeSeriesMatrix` における ICA / PCA 分解や、`Pipeline` クラスを用いた構成が一部実装済み。
- **欠落内容**: これらの統計的手法を実データ（環境ノイズの分離等）に適用する一連のチュートリアル。
- **優先度**: **中 (P1)**

## 3. 「隠れた宝」：テスト済みだが未知の外部連携機能 (Interop)
`tests/interop/` を調査した結果、非常に広範な科学・工学ライブラリとの連携機能が実装済みであることが判明したが、ドキュメント化されていない。

| 分野 | 対象ライブラリ | 検証内容 |
| :--- | :--- | :--- |
| **地球科学** | MetPy, WRF, Harmonica | 気象データ、重力分布、地理座標変換 |
| **電磁場・回路** | eMG3D, SimPEG, PySpice | 3D電磁解析、回路シミュレーション連携 |
| **構造・振動** | OpenSees, Exudyn, SDynPy | 構造成分解析、多体系力学、モーダル解析 |
| **音響・無線** | pyroomacoustics, scikit-rf | 室内音響、RF/マイクロ波解析 |
| **神経科学** | MNE, Neo | EEG/MEG データ構造との変換 |
| **高性能DB** | NetCDF4, Xarray, Zarr, Polars | 大規模多次元データの並列・高速 I/O |

- **提案**: これらを「ドメイン特化型チュートリアル」として整理することで、`gwexpy` の汎用性をアピールできる。

## 4. 未実装の重要機能 (P1 候補)
`STRATEGY.md` および `API_MAPPING.md` で言及され、現場需要が高いが未着手の項目。

1. **インジェクション解析パイプライン (NInjA 互換)**
   - `Bruco` や `estimate_coupling` を統合し、設定ファイルから一括で PEM 注入解析を実行する高水準 API (`InjectionAnalysisPipeline`) が不足。
2. **ADX3 CSV 形式公式リーダー**
   - 現場独自の CSV フォーマットを `TimeSeries.read(format='adx3')` で直接読めるようにする統合。
3. **キャリブレーション・チェーン管理**
   - 周波数依存の応答関数を系統的に管理・適用する `CalibrationChain` の整備。

## 6. docstring の網羅性調査
`ruff` を用いたコードベースの静的解析（D100-D107）の結果、以下の傾向が確認された。

| カテゴリ | 状況 | 主な欠落箇所 |
| :--- | :--- | :--- |
| **コアデータ構造** | 良好 | `TimeSeries`, `FrequencySeries` 等の主要メソッドは概ね記述済み。 |
| **Histogram** | **極めて不十分** | クラス (`D101`)、公開メソッド (`D102`) の多くに docstring がない。 |
| **Interop** | **不十分** | 各種 `to_*`, `from_*` 関数やモジュール全体 (`D100`) の説明が欠落。 |
| **Field** | **不十分** | `VectorField`, `TensorField` の説明が簡素または未記述。 |

- **課題**: `help()` やエディタの補完機能で適切な情報を表示できないため、新規ユーザーの参入障壁となっている。

## 7. 型定義 (mypy) の網羅性調査
`mypy --disallow-untyped-defs` による主要モジュールの走査結果。

| カテゴリ | 状況 | 主な問題点 |
| :--- | :--- | :--- |
| **コアモジュール** | 良好 | `TimeSeries` 等の基盤クラスは型定義が整備されている。 |
| **Histogram** | **注意** | `Any` の多用により、実質的な型チェックが機能していない箇所がある。 |
| **Interop** | **深刻な欠落** | 外部連携関数の多くで引数・戻り値の型指定がなく、実行時までエラーが潜在する。 |

- **具体的な指摘**: `control_.py`, `frequency.py`, `finesse_.py` 等において `[no-untyped-def]` エラーが多数検出された。
- **リスク**: 実行時エラーの潜在、およびエディタ（IDE）による補完・静的解析の恩恵が受けられないことによる開発効率の低下。

## 8. API Reference のカバー率調査
`docs/web/en/reference/api/index.rst` と実際のパッケージ構成を比較した結果、以下のモジュールが API Reference に登録されていない（＝公式ドキュメントに表示されない）ことが判明した。

### 8.1 完全に登録されていないパッケージ
| パッケージ | 重要度 | 理由 |
| :--- | :--- | :--- |
| **`gwexpy.histogram`** | **高** | 統計解析の主軸だが、説明が一切ない。 |
| **`gwexpy.interop`** | **高** | 50以上の外部連携機能（ML、制御系等）が隠れたままになっている。 |
| **`gwexpy.time`** | **高** | GPS時刻変換等の日常的な機能。 |
| **`gwexpy.cli`** | **中** | ツールとしての利用。 |
| **`gwexpy.segments`** | **低** | 指定のデータ領域操作。 |
| **その他** | **低** | `astro`, `detector`, `gui`, `utils`, `log`, `numerics` 等。 |

### 8.2 部分的な未カバー・不整合
- **`gwexpy.signal`**: `spectral` と `preprocessing` は登録されているが、`qtransform.py`, `filter_design.py`, `fft.py`, `window.py` 等のサブモジュールが index から漏れている。
- **`gwexpy.spectral`**: ソース上の公開機能に対し、API Ref に登録されている関数が極めて少ない（3件のみ）。

## 9. ファイル I/O 対応フォーマットの整合性調査
ユーザー向けの [io_formats.md](file:///home/washimi/work/gwexpy/docs/web/ja/user_guide/io_formats.md) および開発者向けの `API_MAPPING.md` と、実際の実装 (`gwexpy/timeseries/io/`) を比較した結果。

- **主要な整合性**: GBD, DTTXML, Seismic (ObsPy), Zarr, NetCDF4 等の独自フォーマットはドキュメント通り実装されている。
- **ドキュメントの更新漏れ**:
    - **`csv_enhanced` (強化版CSV)**: 実装済みだが、どのドキュメントの一覧表にも掲載されていない。
    - **`ndscope_hdf5`**: `API_MAPPING.md` にはあるが、ユーザー向けの `io_formats.md` には記載がない。
- **ステータスの妥当性**: `ADX3` フォーマットが「未実装 (P1)」とされている点は、コードベースにパーサが存在しない実態と一致している。

## 10. テストカバレッジの再確認調査
`pytest-cov` を用いた、重要モジュールのテスト網羅率の再検証結果。

| モジュール | カバレッジ | 評価と課題 |
| :--- | :--- | :--- |
| **`gwexpy/histogram/`** | **95% 以上** | 内部実装は極めて堅牢。ドキュメントを整備すれば即主力機能化が可能。 |
| **`gwexpy/interop/`** | **約 72%** | 全体的に合格点だが、特定の連携モジュールに深刻な未カバーがある。 |
| **`interop/simpeg_.py`** | **8% (深刻)** | 実装はされているが、ほとんどテストが実行されていない。 |
| **`interop/skrf_.py`** | **72%** | 主要な解析パスの一部が未検証。 |
| **`interop/xarray_ / zarr_`** | **90% 前後** | 並列 I/O 関連は良好にテストされている。 |

- **結論**: 「チュートリアルやドキュメントの欠落」には、**「実装は完成しているが紹介されていないだけのもの（Histogram等）」**と、**「実装はされているがテストもドキュメントも不十分で不安定なもの（SimPEG連携等）」**の2種類が混在している。

## 11. 重複コード・レガシーコードの残存調査
`docs_internal/references/SampleCodes_GWpy/` 内のレガシーコードと `gwexpy` の現状機能の比較結果。

- **冗長なコード（削除・アーカイブ推奨）**:
    - **`gbd2gwf.py`**: GBD バイナリのパースと時刻補正を行うスクリプト。リポジトリ内に多数（9件以上）コピーされているが、現在は `TimeSeriesDict.read(format="gbd", timezone="Asia/Tokyo")` で完全に置換済み。
- **有用な参照実装（維持推奨）**:
    - **`read_adx3.py`**: ADX3 CSV 形式の解析ロジックを含む。`gwexpy` での公式リーダー (`format='adx3'`) 実装において、仕様の「正解」として不可欠。
- **課題**: 既に `gwexpy` でスマートに書ける処理に対して、依然として古いスクリプトが使われ続けている箇所があり、ナレッジの断片化が起きている。

## 12. ドキュメントビルドの安定性調査 (静的解析)
 Sphinx の設定 (`conf.py`) および関連コードの解析結果。

- **Mock の多用**: 42 種類以上の外部ライブラリを Mock してビルドを通しており、継承関係の解析が不完全になるリスクがある。
- **遅延インポートによる副作用回避**: `gwexpy/plot/__init__.py` 等で、重い依存関係（`ligo.skymap` 等）を `__getattr__` で遅延読み込みしている。ビルドエラーは防げているが、`autosummary` がクラスを自動検出できない副作用を生んでいる。
- **膨大な警告抑制 (`nitpick_ignore`)**: 200 行以上にわたる無視リストでリンク切れエラーを力技で抑制しており、本来修正すべき重要なエラーが埋もれている。
- **nbsphinx の実行負荷**: ビルド中にノートブックを実行する設定になっており、外部データ取得が必要なコードが含まれる場合、ビルド環境のネットワーク状況によってビルドが失敗する「不確実性」を抱えている。
